# Cache-to-Cache (C2C)

> Fu et al., **ICLR 2026**. arXiv:2510.03215. Tsinghua + Infinigence + CUHK + SJTU + SLAI + Shanghai AI Lab.
> Code: `thu-nics/C2C` (Python package named **`rosetta`**).

---

## Summary

- **Core question**: *Can LLMs communicate beyond text?*
- **Idea**: replace text-to-text (T2T) inter-LLM messaging with **direct KV-cache fusion** between a **Sharer** model and a **Receiver** model.
- **How**: train a small neural **fuser** that projects Sharer KV-cache into Receiver's representation space and combines (residual injection + learnable per-layer gate) into Receiver's own KV-cache. Both LLMs are frozen.
- **Results**: +8.5–10.5% accuracy vs Receiver-only, +3.0–5.4% vs T2T baseline, ~2.0–2.5x speedup vs T2T (avoids generating any intermediate "message" tokens — only Receiver decodes the answer).
- **Why interesting**:
    - **Bandwidth bottleneck of text** is the central pitch — natural language is low-bandwidth, lossy, ambiguous, and sequential (token-by-token decode latency).
    - KV-cache is a **higher-bandwidth medium**: full hidden representations of every layer.
    - Demonstrates that different LLMs' KV-caches are **convertible across families/sizes** via a learned 3-layer MLP (t-SNE shows transformed source cache lands inside target manifold).
    - Connects to **latent reasoning / looped transformers** (e.g., TaH, Saunshi 2025) — even self-fusion (identical Sharer=Receiver) helps.

---

## Motivation / Problem

- **Multi-LLM systems** (collaborative agents like MetaGPT, Chain-of-Agents, MoA, multi-agent debate; routing systems like RouteLLM) currently exchange info via text.
- **3 limitations of T2T**:
    1. **Low bandwidth**: high-dim internal reps compressed to linear strings, then re-encoded by receiver.
    2. **Ambiguity**: idioms, vague refs, underspecified language; rigid templates (MCP, A2A) don't fix open-domain comm.
    3. **Latency**: every exchange = sequential token decode.
- **Motivating example (Fig. 2)**: Coder LLM tells Writer LLM "place after `<p>`". Writer doesn't know `<p>` is a section delimiter — misplaces content. C2C transfers the structural understanding directly.

## Oracle Experiments (motivation for the design)

Two oracle setups establish feasibility before the actual method.

### Oracle 1: Cache Enrichment (Section 3.2.1)
- Q: does richer KV-cache help, independent of context length?
- Setup: compare
    - **Direct**: prefill on $X$ only.
    - **Few-shot**: prefill on $E \oplus X$, decode with full cache → longer cache + richer.
    - **Oracle**: prefill on $E \oplus X$, then **discard the exemplar slice**, keep only the question-aligned cache (same length as Direct but with values "enriched" by attention to exemplars).
- Formally: $\mathcal{C}^*(X) = \mathcal{C}_{[|E|:|E|+|X|]}(E \oplus X)$ — same length, richer semantics.
- **Result (Table 1)**: Direct 58.42%, Few-shot 63.39%, **Oracle 62.34%** → most of the few-shot gain comes from richer cache semantics, not from extra context tokens.
- **Layerwise (Fig. 4 / Table 10)**: only some layers benefit from enrichment; enriching all is suboptimal; enriching worst layers hurts. → motivates **learnable per-layer gate**.

### Oracle 2: Cache Transformation (Section 3.2.2)
- Q: is Sharer's KV-cache *convertible* into Receiver's representation space?
- Train a 3-layer MLP with MSE loss to map Qwen3-4B KV (last layer) → Qwen3-0.6B KV (last layer) on MMLU.
- **t-SNE (Fig. 3)**: raw source/target caches are far apart; **transformed** source lies inside target manifold (but covers a smaller subset → reflects model-specific specialization).
- **Effective rank (Table 2)**: K-cache rank 388 → 395, V-cache 532 → 560 after fusion → semantic content increases.
- Venn diagrams (Fig. 7): different LLMs answer **different subsets** correctly → complementary knowledge.

## Method

### Overview (Section 3.3.1)

- Two frozen models: **Sharer** $\mathcal{S}$ (provides context understanding) and **Receiver** $\mathcal{R}$ (generates the answer).
- **C2C Fuser** $\mathcal{F}_n$: one per Receiver layer $n$; takes Receiver's layer-$n$ cache $\mathcal{C}_n(X)$ and Sharer's mapped layer cache $\mathcal{C}^{\mathcal{S}}_{\mathcal{G}(n)}(X)$, returns the fused cache.
- **Residual integration** (key design choice):

$$\mathcal{C}^{\mathcal{F}} = \{\mathcal{C}_n(X) + \mathcal{F}_n(\mathcal{C}_n(X), \mathcal{C}^{\mathcal{S}}_{\mathcal{G}(n)}(X))\}_{n=1}^{N}$$

- Decoding then uses fused cache: $y_{i+1} = \mathcal{P}(y_i | \mathcal{C}^{\mathcal{F}}(X) \oplus \mathcal{C}(Y_{[0:i]}))$.
- **No intermediate text** generated by Sharer — only KV-caches.

### Fuser Structure (Section 3.3.2)

Three submodules:

1. **Projection module**
    - Concat Receiver KV + Sharer KV along channel.
    - 3-layer MLP (Qwen-style SwiGLU, pre-RMSNorm, residual sublayers) → Receiver's KV dim.
2. **Dynamic weighting module**
    - Input-aware **head modulation** layer.
    - Reweights projected features per-head, per-token.
3. **Learnable gate**
    - Per-layer scalar gate (two scalars: one for K, one for V).
    - **Gumbel-sigmoid with temperature annealing** $T: 1.0 \to 0.001$ → differentiable at train time, near-binary at inference.
    - Decides whether to inject Sharer's contribution at this layer.

> **Why residual** (Section 3.3.2): direct overwrite destroys Receiver's own info. Residual lets Receiver keep its baseline behavior and only get augmented where the gate fires.

### Model Alignment (Section 3.3.3)

Two alignment problems when fusing across model families:

- **Token alignment** (different tokenizers):
    - Decode each Receiver token to string, re-encode with Sharer tokenizer.
    - **1-to-many** case → two strategies: **first-occurrence** vs **maximal-coverage** (decode each candidate, pick longest string). Paper defaults to **maximal-coverage** (~80% identical results across strategies in practice).
    - **Template tokens** (chat role delimiters, etc.) padded to equal length (no semantic content).
- **Layer alignment** (different layer counts):
    - **Terminal alignment** (default): align last-to-last, penultimate-to-penultimate, etc.; shallow layers of the larger model unused.
    - **Depth-normalized**: $j^* = \arg\min_j |i/(L_{\min}-1) - j/(L_{\max}-1)|$ (uniform spread).
    - Terminal chosen empirically (slightly better, simpler).

### Training Scheme (Section 3.3.4)

- **Freeze** Sharer + Receiver; train only fuser parameters.
- **Loss**: standard SFT next-token prediction on Receiver's response, conditioned on **fused** cache.
- Three-stage forward:
    1. Both models prefill the prompt → produce their KV-caches.
    2. Fuser merges → replaces Receiver's cache with fused cache.
    3. Receiver prefills the *response* using fused cache (label tokens supervised); gradients flow back through fuser only.

### Experiments / Results

- **Models**: Qwen3 (0.6B–14B), Qwen2.5 (incl. Coder, Math), Llama3.2 (1B), Gemma3 (1B). Default Receiver = **Qwen3-0.6B**.
- **Benchmarks**: OpenBookQA, MMLU-Redux, ARC-Challenge, C-Eval (Chinese), LongBenchV1 (long context), GSM8K (agentic).
- **Main results** (Table 4): C2C > T2T > routing > individual model across all Sharer choices.
    - e.g., Sharer Qwen3-4B-Base + Receiver Qwen3-0.6B on OpenBookQA: receiver-only 39.20, T2T 46.40 (5.08s), C2C **53.20** (0.34s).
    - Latency table (Table 3 MMLU-Redux): T2T total **1596 ms** (Sharer decodes 80 tokens of background, 1312ms), C2C **445 ms** (90ms KV-cache fusion replaces decoding).
- **Scaling**: gains grow with **stronger Sharer**, modest gains with stronger Receiver. C2C scales similarly across long-context regimes (LongBenchV1 up to 8k+).
- **Heterogeneous + swap** (Table 7): C2C works with Gemma sharer, Qwen2.5-Math sharer, Qwen2.5-Coder sharer; even Sharer/Receiver swap robustly improves (T2T degrades under swap).
- **Ablations**:
    - **Single** fine-tune vs **Identical** (Sharer=Receiver=Qwen3-0.6B) vs **C2C** (different): C2C > Identical > Single → gains aren't just extra parameters. Self-fusion still helps (echoes latent reasoning / looped transformers).
    - **Fuser components** (Table 8): pure projection 20.7%, +fuse (residual, both caches) +24.18%, +gate +3.07% → all three matter.
- **C2C-C variant** (Appendix A.1.3): adds extra projection stage (3-layer MLP to project source cache into receiver dim before concat); recovers up to **86% PGR** (performance gap recovered between Receiver and Sharer accuracies).
- **Gate behavior** (Appendix A.4.2):
    - General-purpose training (OpenHermes 500k) → gates almost always open (~98%); modulation done by dynamic weights.
    - Task-specific training (MMLU auxiliary) → sparse gates (~53% open), weights larger on open layers → narrow specialization.
- **Effective rank** (Section 4.5): both K and V effective ranks rise after C2C → Sharer's semantics get injected.
- **Failure mode** (Appendix A.4.6): weak Sharer with strong Receiver can mislead → accuracy drop (limitation).
- **Multi-sharer extension** (Appendix A.5.1, Table 13): one Receiver, two Sharers → trained pairwise, then both used. Qwen3-0.6B (35.53) + Sharer1 Qwen3-4B-Base + Sharer2 Qwen2.5-1.5B-Math → **64.60%** MMLU (both > either alone).
- **Multi-receiver multi-sharer** (Appendix A.5.2): $O(N)$ scaling via shared latent KV space; M projectors + N fusers instead of $M \times N$ fusers.
- **Agentic flow** (Appendix A.5.3, Table 15): GSM8K problem interpreter + solver. Single model 41.17, T2T 61.18, C2C 62.55, **T-C2C** (T2T interpretation **plus** C2C cache fusion) **78.01** → composable.

### Key Components

#### `C2CProjector` — the actual fuser used in the paper

- **File**: `rosetta/model/projector.py:862-1024`
- **The "concise" projector specialized to the published C2C config**; commented summary at top of class matches paper:
    > *Projections: StandardMLP (pre-RMSNorm, SwiGLU, residual per sublayer). Concat: enabled, followed by linear combiner. Gate: scalar parameter with Gumbel-sigmoid during training. Weights: input-dependent, head_merged granularity using target and projected key. Target preservation: add_self=True, preserve_target_weight=False. Temperatures: annealed gate temperature (1.0 -> 0.001 over 1929 steps), scalar_temperature=1.0.*

- **Architecture** (forward at `projector.py:947-1024`):
    1. **Concat** Sharer and Receiver KV along feature dim: `key_cat = cat([source_key_flat, target_key_flat], dim=-1)`.
    2. **In-projection** `key_in: nn.Linear(in_dim + out_dim, hidden_dim)` (separate for K and V).
    3. **Common embedding** `key_mlp1: RegularMLP(num_layers=1)` — `RegularMLP` is a stack of `StandardFFNLayer` (pre-RMSNorm, GELU-by-default, residual).
    4. **Two heads from common hidden**:
        - **Scalar/weight head**: `key_scalar_mlp2 (num_layers=1)` + `key_scalar_head: Linear(hidden, num_heads)` → per-token per-head scalar (this is the "dynamic weighting" module).
        - **Projection head**: `key_proj_mlp2 (num_layers=num_layers-2)` + `key_proj_out: Linear(hidden, target_dim*num_heads)` → projected K of Receiver shape.
    5. **Gate**: scalar `nn.Parameter` `key_gate_logit` and `value_gate_logit`; **Gumbel-sigmoid** at train time, hard threshold at eval (`projector.py:993-1002`):
       ```python
       g1 = -log(-log(u1 + eps) + eps)               # Gumbel
       key_gate = sigmoid((key_gate_logit + g1) / gate_temperature)
       # at eval:
       key_gate = (key_gate_logit > 0).float()
       ```
    6. **Combination** (`projector.py:1009-1010`):
       ```python
       output_key   = target_key   + key_gate   * sigmoid(key_scalar)   * projected_key
       output_value = target_value + value_gate * sigmoid(value_scalar) * projected_value
       ```
       → **residual injection**, exactly $\mathcal{C}_n + \mathcal{F}_n(\mathcal{C}_n, \mathcal{C}^{\mathcal{S}})$ from paper Eq. 3.
- **Temperature annealing**: `update_temperature(step)` at `projector.py:942-945`:
  $T(s) = T_0 \cdot (T_f / T_0)^{\min(s/S, 1)}$ with $T_0=1.0$, $T_f=0.001$, $S=1929$ steps. Called per optimizer step in training loop (`SFT_train.py:833-837`).

#### `AllInOneProjector` — the "kitchen-sink" projector (for ablations / older variants)

- **File**: `rosetta/model/projector.py:174-770`
- Highly configurable; gate/weight granularity ∈ {`scalar`, `token`, `head`, `head_merged`, `value`}; gate/weight either parameter-based or input-dependent MLP; choice of `preserve_target_weight`, `add_self`, `use_concat`, `use_gumbel`.
- Default config in `recipe/train_recipe/all_in_one.json` uses `gate_granularity="scalar"`, `weight_granularity="head_merged"`, `weight_depends_on_input=True`, `weight_input_features="target_projected_key"`, `preserve_target_weight=False`, `use_swiglu=True`, `use_concat=True` — matches paper's design.
- C2CProjector is essentially a cleaned-up, fixed-config version of this.

#### `RosettaModel` — the wrapper

- **File**: `rosetta/model/wrapper.py:46`
- Holds `model_list = [base_model (Receiver), teacher_model (Sharer), ...]` and `projector_list`.
- `set_projector_config(source_model_idx, source_model_layer_idx, target_model_idx, target_model_layer_idx, projector_idx)` builds nested dict:
  ```
  projector_dict[target_model_idx][source_model_idx][target_layer_idx] = [(source_layer_idx, projector_idx), ...]
  ```
- **`kv_cache_index`** (forward arg, `wrapper.py:350-585`): per-section tensor with shape `(B, sec_seq_len, 2)`. The first element `[i][0][0][0]` is a **bitmask** selecting which Sharers contribute at that section:
    - `-1` = no projection (decode-only / response section).
    - `>0` = bit $i$ selects sharer at `model_list[i+1]`. So `1`=sharer1, `3`=both sharers, `7`=all three.
- **Forward flow**:
    1. Splits input into sections by `kv_cache_index`.
    2. For non-final sections (prompt prefill): runs Receiver forward → caches Receiver KV; runs each Sharer forward (no_grad, eval mode) → caches Sharer KV; if `sharer_mask > 0`, calls projector on each (source_layer, target_layer) pair and **writes fused cache back into Receiver's cache slice** for that section (`wrapper.py:531-572`).
    3. Final section is decoded with the modified cache.
    - **Fusion modes**: `"sequential"` (each source updates base cache iteratively) or `"parallel"` (each source projects from clean base cache, sum residuals at end). Default `sequential`.
- **`include_response` mode** (`wrapper.py:280-343`): also fuses into the response section. Uses an interesting trick — **monkeypatches Qwen3 attention** (`_monkeypatch_qwen3_attention_forward`) so the modified KV is used **in the same forward pass** (not just the next token). Clean restore via `remove_hooks`.
- **Custom `generate`** (`wrapper.py:587-816`): does not use HF `generate`. Prefill via own `forward`, then loop: sample → append → decode one token with `kv_cache_index=[[-1,0]]` (no projection during decode). Supports temperature/top-p/top-k, repetition/presence/frequency penalty, streamer, EOS handling.

#### `generate_kv_cache_index` — how training builds the index

- **File**: `rosetta/train/dataset_adapters.py:480-498`
- Maps instruction tokens to bitmask `1` (project sharer 1) for all but last instruction token, and response tokens to `-1` (no projection):
  ```python
  instruction_index = torch.tensor([1, 0]).repeat(instruction_length - 1, 1)
  label_index       = torch.tensor([-1, 0]).repeat(full_length - instruction_length + 1, 1)
  ```
- So during training the fuser only runs on the prompt's prefill cache; response tokens are generated using the (already fused) cache without further fusion.

#### `TokenAligner` — cross-tokenizer alignment

- **File**: `rosetta/model/aligner.py:20`
- Decode each SLM (receiver) token → re-encode with LLM (sharer) tokenizer; cache mappings.
- Two **strategies** via `AlignmentStrategy` enum: `FIRST` (take first sharer token in many-to-one) and `LONGEST` (maximal-coverage — paper default).
- Special-token mapping via `_map_special_token`.
- Returns aligned LLM ids + per-section breakdown (template vs message sections).

#### Layer mapping strategies

- **File**: `rosetta/train/model_utils.py`
- `last_aligned_sources(num_target_layers, num_source_layers, k=1)`: terminal alignment — target layer $t$ → source layer $offset + t$ (offset = `n_source - n_target`). Anchors from last, walks back. Paper default.
- `k_nearest_sources(...)`: depth-normalized alignment — sort source layers by abs-distance in `[0,1]` to target's normalized position.

#### Token-level evaluation pipeline

- **File**: `script/evaluation/unified_evaluator.py`
- Dataset registry includes MMLU-Redux, MMMLU, GPQA (and more, with subject lists + MMLU subcategory taxonomy).
- Pulls Rosetta model via `rosetta.utils.evaluate.load_rosetta_model`, applies generation config, calls `model.generate` with explicit `kv_cache_index`.
- Two-stage T2T baseline lives in `rosetta/baseline/multi_stage.py:TwoStageInference` — uses `background_prompt` template (e.g., *"Briefly describe the most useful background to solve the problem: {question}"*) to get sharer's textual background, concats with question, feeds to receiver.

### Training / Inference Flow

**Training (`script/train/SFT_train.py:main`)**

1. Load both models in bf16, frozen (`freeze=["teacher","base"]` in config).
2. Compute `base_dim = k_proj.out_features / num_key_value_heads` (per-head dim) for each model → fuser knows dims.
3. Instantiate **one fuser per Receiver layer** (`num_projectors = base_model.config.num_hidden_layers`). Projector type from `projector["type"]`; shared across sources (only M projectors, one per target layer).
4. Build `RosettaModel(model_list=[base, teacher], base_model_idx=0, projector_list=...)`.
5. Apply layer mapping (`last_aligned` or `k_nearest`).
6. Build dataset (default `OpenHermesChatDataset`, 500k samples, `max_word_count=2048`); use `RosettaDataCollator` to handle per-model aligned inputs and kv_cache_index sections.
7. **Optimizer** with three param groups (gate / key_weight+value_weight / other) all at `lr=1e-4`; `AdamW`, weight_decay=0.01, max_grad_norm=1.0, linear scheduler with 10% warmup, gradient_accumulation_steps=8, per-device batch 4, 8 GPUs → effective batch 256.
8. Standard SFT loss (HF `outputs.loss` on labels with `-100` masked instruction tokens). Loss is computed via `RosettaModel.forward(..., labels=...)` which calls the base model with the fused cache.
9. After each optimizer step, call `proj.update_temperature(global_step)` to anneal the Gumbel temperature.
10. Save per-projector `.pt` state_dict + `.json` constructor config (`save_projector` uses captured init args via `@capture_init_args` decorator); save `projector_config.json` (the nested mapping dict).

**Inference (`generate` in `RosettaModel`)**

1. User builds `kv_cache_index` matching prompt vs response sections. Typical: `[1, 0]` for prompt, `[-1, 0]` for response (see README example).
2. Prefill: both models forward → fuser injects → Receiver cache updated.
3. Token-by-token decode loop uses only Receiver (sharer cache is no longer touched after prefill); fast — no sharer decoding at all (paper Table 3: T2T sharer decode 1312 ms → C2C 0 ms; only +90 ms for fusion).

### Code-vs-Paper Notes / Non-obvious Bits

- **"Teacher" = Sharer**, **"base" = Receiver** in code (inverted terminology from paper; this is because the receiver is what the wrapped model returns, hence "base model"). Worth keeping in mind reading configs.
- **`C2CProjector` differs subtly from paper diagram (Fig. 5)**:
    - Paper Fig. 5 shows projection → dynamic weighting → feature fusion → gate → residual add to Receiver.
    - Code path is: shared concat-then-MLP trunk → **two parallel heads**: a projection head (produces Sharer-derived feature in Receiver dim) and a scalar head (per-head sigmoid weight). Then `output = receiver + gate * scalar * projected`. The "feature fusion" step is implicit in concatenating both caches at the input.
- **Gate is scalar per K/V per layer** (just two learnable floats per projector, before Gumbel-sigmoid). This is much coarser than the per-token/per-head granularity options in `AllInOneProjector`. Paper's gating behavior analysis (Appendix A.4.2) is mostly about `AllInOneProjector` with input-dependent gates; the published headline `C2CProjector` uses purely scalar gates and pushes input-dependence into the **dynamic weights**.
- **No-gradient on Sharer**: in `RosettaModel.forward` (`wrapper.py:491-499`) Sharer runs under `torch.no_grad()` and is forced to `.eval()` to disable gradient checkpointing. Only the base (Receiver) model runs with grad enabled — but Receiver is also frozen during training; the only trainable params are the fusers. So in practice the grad-on path is: fuser → Receiver final layers → loss.
- **`include_response`** monkeypatches Qwen3 attention to inject the fused K/V *in the same forward pass* before the cache update — clever workaround to avoid modifying transformers internals. Not used in default training (`include_response=False`).
- **Decode-time `kv_cache_index`** is hardcoded to `[-1, 0]` in `generate` (`wrapper.py:780`) — meaning no further fusion during decoding. Consistent with paper's design: fuse only at prefill, decode is pure Receiver.
- **Multi-source fusion mode**: code supports both **sequential** (cascade fusers) and **parallel** (sum residuals from clean base) — Appendix A.5.1 multi-Sharer experiments use this. Default is `sequential`.
- **`@capture_init_args` + `@register_model`** decorators in `rosetta/utils/registry.py` save constructor kwargs alongside weights → `load_projector` reconstructs the right class from JSON without manual config plumbing. Standard registry pattern.
- **No explicit gate regularization in loss** — the commented-out block in `SFT_train.py:445-449` shows the authors *experimented* with adding `0.0025 * gate` to the loss but ship without it. Gate openness emerges from the SFT loss alone.
- **MMLU vs OpenHermes training**: paper says general-purpose (OpenHermes) → broad gates; task-specific (MMLU aux) → sparse gates. Both correspond to different `anneal_steps` in recipes (1929 for general, 112 for short MMLU run).
- **Live demo and multi-sharer support** in `script/playground/live_chat_example.py` and `gradio_demo.py` — uses `kv_cache_index` bitmask to dynamically select which sharer(s) contribute on a per-query basis.
